# 权限模型

## 1. 设计目标

Agentum 的权限设计需要覆盖平台管理、资产治理、工作流执行和外部系统调用。

要求：

- 权限从第一版开始内建，不后补。
- 角色用于控制人，资源权限用于控制资产。
- MCP、凭证、生产交付节点必须重点保护。
- 所有敏感操作必须审计。

## 2. 权限模型

推荐采用：

```text
RBAC + 资源级权限 + 审计日志
```

RBAC 负责角色，资源级权限负责具体资产。

## 3. 推荐角色

| 角色 | 说明 |
| --- | --- |
| 系统管理员 | 管理全局配置、用户、空间、模型、系统级 MCP |
| 空间管理员 | 管理业务空间成员、资产和权限 |
| 流程设计者 | 创建和编辑工作流 |
| 智能体管理员 | 创建、维护和发布智能体模板 |
| 能力管理员 | 管理 Skills、MCP、知识库和工具凭证 |
| 审核人 | 处理人工审核节点 |
| 执行人 | 触发流程、填写输入、查看自己的运行结果 |
| 观察者 | 只读查看流程、资产和执行记录 |

## 4. 资源类型

- Workflow

- WorkflowRun
- NodeRun
- AgentTemplate
- Skill
- McpService
- KnowledgeBase
- ToolCredential
- DeliveryTarget
- AuditLog
- User
- Space

## 5. 权限动作

基础动作：

- `read`
- `create`
- `update`
- `delete`
- `execute`
- `publish`
- `approve`
- `cancel`
- `retry`
- `manage_permission`

敏感动作：

- `manage_credential`
- `invoke_mcp`
- `invoke_sensitive_mcp`
- `download_file`
- `deliver_to_external_system`
- `view_audit_log`

## 6. 权限判断顺序

建议按以下顺序判断：

1. 用户是否登录。
2. 用户是否属于当前空间。
3. 用户角色是否具备基础动作。
4. 用户是否对目标资源有访问权限。
5. 操作是否涉及敏感能力。
6. 是否需要二次确认或人工审批。
7. 记录审计日志。

## 7. MCP 权限

MCP 是高风险能力，需要单独控制。

每个 MCP 服务应配置：

- 可见范围
- 可调用角色
- 可调用智能体
- 可调用工作流
- 参数白名单
- 调用频率限制
- 是否需要审批
- 是否允许生产环境调用

## 8. 凭证权限

工具凭证不允许前端直接读取明文。

要求：

- 凭证服务端加密存储。
- 前端只显示脱敏信息。
- 凭证绑定、解绑、更新必须审计。
- 生产凭证和测试凭证隔离。
- 高风险凭证需要空间管理员审批。

## 9. 审计日志

必须审计：

- 登录和登出
- 创建、修改、删除工作流
- 发布工作流
- 修改智能体提示词
- 修改 Skill 或 MCP 配置
- 调用 MCP
- 绑定或更新凭证
- 人工审核通过或驳回
- 对外交付文件或消息
- 下载敏感文件

审计字段：

- 操作人
- 操作时间
- 资源类型
- 资源 ID
- 动作
- 请求 ID
- IP
- User-Agent
- 操作前摘要
- 操作后摘要
- 结果状态
