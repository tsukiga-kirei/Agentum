# 产品界面分区

## 1. 设计目标

Agentum 不应该让所有用户都进入同一个画布界面。

不同角色关注点不同：

- 业务人员只关心“我要完成什么工作、现在轮到我做什么、结果在哪里”。
- 流程设计者关心“流程怎么编排、节点怎么配置、变量怎么传递”。
- 能力管理员关心“Skills、MCP、智能体模板、知识库如何治理”。
- 系统管理员关心“权限、凭证、审计、系统配置是否安全可控”。

因此前端需要明确区分：

```text
认证与账号区
业务使用区
流程设计工作台
能力管理区
系统管理区
运行审计区
```

这些区域可以在同一个前端应用 `apps/web` 中实现，但路由、导航、权限和页面心智必须分开。

## 2. 认证与账号区

认证与账号区面向所有用户，是进入平台前和管理个人身份的基础区域。

目标：

- 处理登录、会话过期、无权限、退出登录。
- 展示和维护个人资料、账号安全、默认组织、默认空间和当前角色。
- 让用户清楚知道自己当前以哪个身份在操作。

### 2.1 典型页面

- 登录
- SSO / MFA 验证
- 会话过期
- 无权限
- 个人资料
- 账号安全
- 当前组织 / 空间 / 角色切换

### 2.2 页面重点

认证与账号区应展示：

- 当前登录用户
- 当前组织
- 当前空间
- 当前角色
- 账号状态
- 安全设置入口

不展示：

- 未授权的组织成员列表
- 工具凭证
- MCP 参数
- 审计日志明细

## 3. 业务使用区

业务使用区面向普通业务人员、流程发起人、审核人和执行人。

目标：

- 不暴露复杂画布。
- 不要求用户理解智能体、MCP、变量、节点配置。
- 让用户像使用业务系统一样完成任务。

### 3.1 典型页面

- 我的待办
- 我发起的流程
- 可用流程模板
- 发起流程
- 填写资料
- 追问与确认
- 人工审核
- 运行进度
- 交付物查看

### 3.2 页面重点

业务人员进入系统后，优先看到：

- 当前待处理事项
- 可发起的业务流程
- 流程当前进度
- 需要填写或确认的内容
- 最终生成的报告、文件、邮件或 OA 流程

业务使用区不默认展示画布。对于复杂流程，可以展示简化版步骤条。

步骤条示例：

```text
提交资料 -> AI 分析 -> 补充确认 -> 报告生成 -> 人工审核 -> 完成交付
```

### 3.3 业务态流程详情

业务态流程详情应展示：

- 当前步骤
- 当前状态
- 正在等待谁
- 用户需要做什么
- 已完成步骤
- AI 已完成的工作摘要
- 可查看的输出结果
- 最终交付物

不展示：

- 节点坐标
- 节点连接线
- 原始 Prompt
- 未授权 MCP 配置
- 复杂变量配置

## 4. 流程设计工作台

流程设计工作台面向流程设计者、智能体管理员和高级配置人员。

目标：

- 提供画布。
- 支持节点编排。
- 支持智能体、Skills、MCP、变量、条件、交付配置。
- 支持测试运行和发布校验。

### 4.1 典型页面

- 工作流列表
- 工作流编辑器
- 节点配置面板
- 变量面板
- 测试运行
- 发布校验
- 版本记录

### 4.2 页面重点

流程设计者需要看到：

- 节点类型
- 节点连接关系
- 输入变量
- 输出变量
- 智能体装配
- Skills / MCP 配置
- 用户输入配置
- 并行与合并策略
- 条件分支规则
- 交付目标

这里才是完整画布的主要使用场景。

## 5. 能力管理区

能力管理区面向能力管理员和智能体管理员。

目标：

- 管理可复用的原子能力。
- 控制版本、权限和发布状态。
- 避免流程设计者重复创建不受控能力。

典型页面：

- 智能体模板库
- Skills 库
- MCP 服务库
- 知识库
- 模型供应商
- 交付目标
- 工具凭证

每个能力资产应展示：

- 名称
- 描述
- 版本
- 状态
- 适用范围
- 权限
- 最近更新时间
- 被哪些工作流引用

## 6. 系统管理区

系统管理区面向系统管理员和空间管理员。

典型页面：

- 用户管理
- 角色管理
- 组织管理
- 空间管理
- 成员管理
- 邀请管理
- 权限策略
- 系统配置
- 凭证管理
- 安全策略
- 集成配置

系统管理区应严格控制访问权限，不应该出现在普通业务人员导航中。

系统管理区必须回答：

- 当前组织有哪些用户。
- 每个用户属于哪些空间。
- 每个用户在每个空间拥有哪些角色。
- 哪些角色能操作工作流、资产、MCP、凭证和审计日志。
- 哪些敏感操作需要审批或二次确认。

## 7. 运行审计区

运行审计区面向管理员、审核人、流程负责人和问题排查人员。

典型页面：

- 工作流运行记录
- 节点运行详情
- 工具调用记录
- 变量快照
- 用户输入记录
- 人工审核记录
- 交付记录
- 错误与重试记录

运行审计区可以从业务流程详情进入，也可以从管理后台集中查询。

## 8. 不同角色的默认入口

| 角色 | 默认入口 | 是否默认看画布 |
| --- | --- | --- |
| 未登录用户 | 登录页 | 否 |
| 业务人员 | 我的待办 / 可用流程 | 否 |
| 流程发起人 | 发起流程 / 我的流程 | 否 |
| 审核人 | 待审核事项 | 否 |
| 流程设计者 | 工作流设计器 | 是 |
| 智能体管理员 | 智能体模板库 | 可选 |
| 能力管理员 | Skills / MCP 管理 | 否 |
| 系统管理员 | 系统管理区 | 否 |
| 审计人员 | 运行审计区 | 否 |

## 10. 推荐前端代码分层

前端代码可以按“产品区域 + 业务能力”组织。

```text
apps/web/src/
  app/
  routes/
  surfaces/
    auth/
    account/
    workbench/
    designer/
    assets/
    admin/
    audit/
  features/
    auth/
    account/
    organization/
    workflow/
    agent/
    assets/
    runs/
    permission/
    audit/
  components/
  services/
  stores/
  lib/
  types/
```

说明：

- `surfaces` 表示用户进入系统后看到的产品区域。
- `features` 表示可复用业务能力。
- 业务使用区可以复用 `runs`、`workflow` 的只读能力，但不暴露完整画布编辑能力。
- 流程设计工作台可以复用所有 workflow 设计能力。

## 11. 与后端的关系

这里的“工作台 / 管理区”是产品界面概念，不等同于后端服务。

后端仍然是统一的 `apps/api`，通过权限控制决定不同角色能访问哪些资源和动作。

关键原则：

- 前端隐藏入口只是体验优化，后端必须做真实权限校验。
- 业务人员不能通过直接调用接口获得未授权 Prompt、MCP 凭证或节点配置。
- 流程运行结果可以向业务人员展示摘要，但敏感工具参数必须脱敏。
- 登录态、组织归属、空间角色和资源权限必须由后端签发和校验，前端不能自行构造。

## 12. 设计结论

Agentum 至少需要三套清晰心智：

- **身份与权限心智**：我是谁，属于哪个组织和空间，以什么角色操作。
- **业务使用心智**：我现在要处理什么，下一步怎么做，结果在哪里。
- **流程设计心智**：流程如何编排，智能体如何装配，变量如何流转。

这些心智不能混在一个默认入口里。普通业务人员不应该被画布、变量和工具配置打扰；未授权用户也不应该看到任何内部业务入口。
