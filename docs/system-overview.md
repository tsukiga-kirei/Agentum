# 系统详细梳理介绍

## 1. 产品定位

Agentum 是一个以智能体为载体的企业工作流平台。

它和常见 AI 聊天式工作流的关键区别是：用户不是在一个聊天框里等待 AI 一次性完成所有事，而是在一条明确的业务流程中，按步骤完成输入、确认、追问、审核、回退、暂停、恢复和交付。每个节点负责具体任务，每个智能体节点只在明确上下文里使用被授权的 Skills、MCP、提示词模板和模型配置。

核心目标：

- 让企业 SOP 可以被拆成可执行、可暂停、可审计的步骤。
- 让智能体能力可复用、可授权、可版本化。
- 让并行子智能体加快数据获取和分析，再由合并节点组装结果。
- 让最终结果能落到 Word / PDF、邮件、OA、IM、Webhook 或数据库等交付目标。
- 让业务用户看懂自己正在做哪一步，而不是被画布、变量和工具配置打扰。

第一阶段暂不推进知识库 / RAG 作为核心功能。后续如果业务需求变重，再作为能力资产补回，并单独处理版本、权限、引用来源和审计。

## 2. 核心概念

| 概念 | 说明 |
| --- | --- |
| 租户 | 平台最高业务隔离边界，承载租户级模型额度、可用 MCP、可用 Skills、交付能力和系统策略 |
| 组织 / 部门 | 租户内人员组织结构，用于成员管理、待办分派、审核范围和数据筛选 |
| 空间 | 租户内资源协作边界，可用于区分业务线、项目组或环境 |
| 用户 | 登录主体，一个用户可以在不同租户或空间拥有不同角色 |
| 角色 | 动作能力集合，例如执行人、审核人、流程设计者、能力管理员、空间管理员、系统管理员 |
| 资源权限 | 控制具体工作流、资产、运行记录、凭证、审计日志是否可读、可编辑、可执行、可发布 |
| 智能体模板 | 可复用的智能体配置，包含系统提示词、Skills、MCP、模型、输出 Schema 和交互模式 |
| Skill | 提示词和方法论能力，例如需求拆解、追问澄清、风险识别、报告撰写 |
| MCP | 外部系统、工具或数据源连接能力，例如文件读取、数据库只读查询、邮件发送、OA 创建流程 |
| 提示词模板 | 可复用的 Prompt 片段或业务话术模板，供智能体模板或节点引用 |
| 交付能力 | 生成文件、发送邮件、创建 OA、推送 IM、Webhook、写数据库等最终动作 |
| 工作流定义 | 设计态流程模板，包含节点、边、变量和节点配置 |
| 工作流运行 | 某一次实际执行实例，保存状态、变量快照、节点记录、暂停事件和交付记录 |

## 3. 产品区域

Agentum 不应该让所有用户进入同一个默认画布。不同角色看到不同入口和操作。

| 区域 | 面向角色 | 主要内容 |
| --- | --- | --- |
| 认证与账号区 | 所有人 | 登录、会话过期、无权限、个人资料、账号安全、当前租户 / 空间 / 角色 |
| 业务工作台 | 业务人员、执行人、审核人 | 我的待办、可用流程、我发起的流程、运行详情、待办处理、交付物 |
| 流程设计 | 流程设计者、智能体管理员 | 工作流列表、画布、节点配置、变量、测试运行、发布校验 |
| 能力资产 | 能力管理员、智能体管理员 | 智能体模板、Skills、MCP、提示词模板、交付能力资产 |
| 运行监控 | 流程负责人、空间管理员 | 运行状态、失败重试、取消、补偿和管理员介入 |
| 运行审计 | 审计人员、管理员、流程负责人 | 只读运行记录、节点记录、变量快照、工具调用、审核记录、交付记录 |
| 权限管理 | 空间管理员、系统管理员 | 用户、部门、角色、资源权限、敏感动作控制 |
| 系统管理 | 系统管理员 | 租户、模型供应商、全局 MCP / Skills 开关、交付能力、系统参数、多租户策略 |

前端隐藏入口只是降低干扰，不是安全边界。所有接口、资源访问、发布、执行、MCP 调用、凭证和交付动作都必须由后端重新校验。

### 3.1 登录与租户选择

登录页需要同时明确“用户要以什么入口进入”和“非系统管理员归属哪个租户”。租户是平台最高业务隔离边界，业务用户、审核人、流程设计者、能力管理员和空间管理员都必须在登录时选择租户，后端据此校验用户是否属于该租户、是否具备所选入口对应角色，并把当前租户写入会话或 JWT 的活跃角色上下文。

系统管理员入口不绑定某个租户。系统管理员登录后进入平台级系统管理，再在具体租户管理、租户授权、租户统计或跨租户诊断场景中选择目标租户。这样可以避免把平台级操作误认为某个租户内操作。

第一阶段可参考 AuraOA 的登录链路：公开接口返回活跃租户轻量列表，登录请求携带 `tenantId` 和 `preferredRole`；后端签发令牌时写入 `activeRole`、`tenantId`、`spaceId` 和角色集合。公网 SaaS 场景下，公开租户列表可能暴露租户名称，后续可改为租户编码输入、企业域名识别或子域名识别。

### 3.2 页面抽象原则

当前前端页面是为了降低用户理解成本而抽象出的产品视图，不等同于稳定的领域模型、数据库边界或最终信息架构。

页面可以随着开发推进继续拆分、合并或改名。判断一个页面是否应该保留，优先看它是否服务某类角色的具体任务，而不是看它是否刚好对应一个后端表或一个概念名。

设计原则：

- 页面是角色任务入口，领域对象是后端事实来源。
- 页面可以聚合多个底层对象，例如运行审计页可以聚合运行记录、节点记录、等待事件、审计日志、变量快照、MCP 调用和交付记录。
- 页面不应制造新的权责边界。业务处理、运行监控和运行审计应分别对应不同操作权限。
- 过渡期为了说明概念出现的面板或卡片，如果后续被具体模块覆盖，应及时移除，避免形成重复抽象。
- 产品文案优先使用业务语言，技术动作码、表名和事件名只作为次级信息或开发调试信息。

## 4. 业务工作台说明

业务工作台是普通用户的默认入口，不默认展示画布。

### 4.1 我的待办

“我的待办”表示流程已经暂停，并且下一步需要当前用户、当前角色或当前部门处理。

典型待办：

- 用户输入节点等待发起人补充资料。
- 智能体追问确认等待用户回答或确认。
- 人工审核节点等待审核人通过、驳回或修改后继续。
- 高风险 MCP 调用等待审批。
- 交付节点等待交付确认。

待办的核心问题是：现在轮到谁做什么。

### 4.2 运行态摘要

“运行态摘要”表示用户有权限查看的流程运行位置和状态，不一定需要用户处理。

典型状态：

- 运行中：节点正在执行。
- 已暂停：等待某人输入、审核、审批或外部回调。
- 已完成：可以查看交付物。
- 已失败：可以查看错误和是否允许重试。

运行态摘要的核心问题是：流程现在在哪里，为什么。

### 4.3 可用流程模板

可用流程模板是业务用户可以发起的已发布流程。点击后不是进入设计画布，而是创建一次工作流运行，并进入业务态交互页面。

业务态交互页面应支持：

- 按步骤填写资料。
- 查看智能体已完成的摘要。
- 对智能体结果追问或要求改写。
- 确认后进入下一步。
- 流程暂停、退出后恢复。
- 有权限时回退到上一步或取消本次运行。
- 查看最终交付物。

### 4.4 业务运行详情

业务运行详情承接“我的待办”和“运行态摘要”的下一层页面。它面向发起人、当前待办处理人和流程负责人，不面向纯审计人员。

业务运行详情可以出现处理动作：

- 提交用户输入。
- 回答智能体追问。
- 确认或改写智能体输出。
- 人工审核通过、驳回或修改后继续。
- 审批高风险 MCP 调用。
- 确认交付。

这些动作本质上都是“恢复暂停点”，后端必须记录恢复事件、操作人、意见、附件和来源渠道，并重新校验权限。

## 5. 流程设计节点

工作流由固定节点类型组成。固定节点降低复杂度，能力复用来自节点引用的资产。

| 节点 | 作用 | 是否可暂停 |
| --- | --- | --- |
| 触发节点 | 手动、定时、Webhook 或外部事件触发流程 | 否 |
| 用户输入节点 | 表单、文件上传、人员选择、部门选择、补充说明 | 是 |
| 智能体节点 | 调用智能体模板执行分析、生成、判断、提取或追问 | 可选 |
| 并行节点组 | 同时启动多个子智能体或工具任务，加快数据获取 | 取决于子节点 |
| 合并 / 组装节点 | 汇聚并行结果，透传、JSON 合并、模板组装或调用组装智能体 | 可选 |
| 条件分支节点 | 基于变量、规则或外部返回值决定路由 | 否 |
| 人工审核节点 | 人工确认、修改、通过、驳回、回退 | 是 |
| 交付节点 | 生成文件、发送邮件、创建 OA、推送 IM、Webhook、写数据库 | 否 |
| 结束节点 | 结束本次运行并固化结果 | 否 |

并行节点的本意是放入多个子智能体或工具任务并发执行，例如“资料核验智能体、外部数据查询智能体、风险补充分析智能体”同时运行。合并节点负责把它们的输出组装为下游可用变量，必要时可以绑定“报告组装智能体”。

输入节点、条件节点、结束节点这类不依赖智能体的通用节点，建议平台内置。是否也作为“能力资产”管理，后续可根据复用和版本治理需要再决定；第一阶段先按内置节点处理更简单。

## 6. 能力资产

能力资产是流程设计的复用来源。流程节点不复制资产内容，而是引用资产版本。

第一阶段能力资产范围：

- 智能体模板。
- Skills。
- MCP。
- 提示词模板。
- 交付能力资产。

暂不展示知识库资产。

能力资产分为两层：

- 源码 / 实现层：放在仓库 `capabilities/` 目录下，保存产品运行时 Skills、自研 MCP Server、提示词模板源码和交付适配实现。
- 治理 / 授权层：放在数据库和管理页面中，保存资产名称、版本、发布状态、引用关系、租户授权、角色权限和审计策略。

这两层不能混为一个概念。`capabilities/skills/` 和 `capabilities/mcp-servers/` 负责让能力能被开发、测试和部署；能力资产页负责让这些能力被登记、发布、授权、停用和追溯。

`.codex/skills/` 是开发本项目时给 Codex 使用的本地辅助技能，不是 Agentum 产品运行时的 Skill 资产来源。

能力资产页面应聚焦“资产是什么、版本是什么、能被哪些流程引用”。权限策略、敏感审计、租户授权和凭证策略不建议在这里作为主要展示内容，应放到权限管理和系统管理中，以免能力资产页变成安全配置页。

能力资产页不建议展示“流程导入建议”这类设计指导。流程节点如何引用资产，应放在流程设计的节点配置面板中。能力资产页应提供新增、发布、停用、版本管理和查看引用关系等入口。

如果每类资产模块已经提供独立新增入口，就不应再额外保留一个泛化的“资产创建入口”。右侧或辅助区域更适合展示待治理资产、引用关系、缺失授权、草稿待发布和高风险能力缺少审批策略等运营信息。

### 6.1 交付能力资产

交付能力也应成为可治理资产。

示例：

| 交付能力 | 说明 | 风险 |
| --- | --- | --- |
| Word / PDF 生成 | 根据模板和变量生成正式文档 | 中 |
| 邮件发送 | 发送正文和附件，可接 Mailpit 本地验证 | 高 |
| OA 流程创建 | 创建审批或归档流程 | 高 |
| IM 推送 | 推送飞书、钉钉、企微消息 | 中 |
| Webhook 回调 | 回调外部系统 | 中 |
| 数据库写入 | 写入业务表或结果表 | 高 |

高风险交付能力必须支持审批、二次确认、审计和失败重试。

## 7. 系统管理与租户授权

系统管理是平台级入口，负责“哪些底层能力可以被哪个租户看到和使用”。

推荐链路：

```text
系统管理员注册底层能力
  -> 授权给某个租户
  -> 租户 / 空间管理员在租户内分配给角色、部门、智能体或工作流
  -> 工作流运行时根据用户、租户、角色、资源权限再次校验
```

系统管理负责：

- 租户管理。
- 模型供应商和模型配置。
- 全局 MCP 注册、测试、启停。
- 全局 Skills 和提示词模板上架。
- 全局交付能力注册。
- 租户级能力开关和配额。
- 凭证策略和安全策略。
- 多租户隔离、资源配额、数据保留策略。

租户或空间内管理负责：

- 用户、部门、角色和成员。
- 当前租户可用能力的具体授权。
- 哪些角色可以设计流程、执行流程、审核、发布、查看审计。
- 哪些智能体或工作流可以调用某个 MCP 或交付能力。
- 哪些用户、部门或角色可以查看某个智能体、流程运行、交付物或审计记录。

AuraOA 可参考的部分是：系统管理员路由和租户管理员路由分层、JWT + 租户上下文 + 角色校验、部门 / 角色 / 成员管理、租户模型和系统配置管理。Agentum 不照搬实现语言，但保留这种治理边界。

## 8. 权限管理

权限模型采用：

```text
认证身份 + 租户上下文 + 组织 / 部门 / 空间归属 + RBAC + 资源级权限 + 敏感动作控制 + 审计
```

推荐角色：

| 角色 | 说明 |
| --- | --- |
| 系统管理员 | 管理全局配置、租户、模型、全局 MCP、交付能力 |
| 租户管理员 | 管理租户内用户、部门、角色、空间和可用资产 |
| 空间管理员 | 管理某个空间内成员、资产授权和发布策略 |
| 流程设计者 | 创建和编辑工作流、配置节点、测试运行 |
| 智能体管理员 | 创建、维护和发布智能体模板 |
| 能力管理员 | 管理 Skills、MCP、提示词模板和交付能力资产 |
| 审核人 | 处理人工审核节点和高风险审批 |
| 执行人 | 发起流程、填写输入、追问确认、查看自己的结果 |
| 观察者 | 只读查看被授权流程、资产或运行记录 |

基础动作：

- `read`
- `create`
- `update`
- `delete`
- `execute`
- `publish`
- `approve`
- `cancel`
- `retry`
- `manage_permission`

敏感动作：

- `manage_tenant`
- `manage_user`
- `manage_role`
- `manage_model`
- `manage_credential`
- `invoke_mcp`
- `invoke_sensitive_mcp`
- `deliver_to_external_system`
- `download_sensitive_file`
- `view_audit_log`

产品界面不应把这些动作码作为主要文案直接展示给业务管理员。页面应使用中文业务权限，例如“查看、发起流程、编辑配置、发布、审核、重试、取消、查看审计”，动作码只作为后端策略标识或调试信息保留。

权限管理建议按页签组织：

- 人员组织：用户、部门、空间成员关系。
- 角色权限：角色能做哪些业务动作。
- 资源授权：具体流程、智能体、运行记录、交付物授权给哪些用户、部门或角色。
- 敏感动作：高风险 MCP、外部交付、敏感文件下载、凭证申请和审批。
- 审计可见性：谁可以查看只读证据链，以及哪些字段需要脱敏。

凭证配置和凭证明文读取不属于权限管理页的主要职责。系统级凭证策略、模型密钥和全局 MCP 凭证放在系统管理；权限管理只决定谁可以申请、审批或使用这些能力。

## 9. 运行态交互

运行态是 Agentum 的核心体验，应围绕“当前步骤”而不是“聊天历史”组织。

必须支持：

- 暂停：等待用户输入、追问、人工审核、外部回调或高风险审批。
- 恢复：提交表单、确认结果、审核通过、审批工具调用后继续。
- 追问：用户围绕当前智能体结果继续问，确认后写入最终变量。
- 改写：用户要求智能体重写当前节点输出，但仍受输出 Schema 约束。
- 回退：有权限时退回到上一个可回退节点，并保留审计记录。
- 退出：用户离开页面后可从待办或运行记录恢复。
- 重试：节点失败后按策略重试，并记录重试原因和结果。

运行详情应展示：

- 当前状态。
- 当前节点。
- 等待对象。
- 暂停原因。
- 输入变量和输出变量。
- 已完成步骤摘要。
- 工具调用摘要。
- 审核记录。
- 交付记录。
- 最终交付物。

业务用户看到简化步骤条，设计者和审计人员可进入更完整的画布和证据链。

### 9.1 审核、监控与审计边界

“审核”和“审计”必须分开：

- 审核是业务流程中的人工节点，由审核人或流程负责人决定通过、驳回、修改后继续。
- 运行监控是管理员视角，用于查看运行状态、处理失败重试、取消和补偿，所有介入动作都必须写入审计。
- 运行审计是只读证据链，用于还原流程当时发生了什么，不推动业务流程继续。

运行审计可以展示执行链路、节点输入输出快照、变量快照、MCP 调用摘要、模型输出解析、审核记录和交付记录。它们的意义是追溯“结论从哪里来、谁在什么时候做了什么、敏感数据是否脱敏、工具调用是否越权”，不是让审计人员修改当前流程。

“执行链路”是聚合视图，不是新的独立数据源。它应由 `WorkflowRun`、`NodeRun`、`WaitingEvent`、`RunEvent` / `audit_logs`、`mcp_call_logs`、`delivery_records` 和 `variable_snapshots` 组合生成。审计日志是证据来源之一，但不应被设计成承载所有执行状态的万能流水表。

暂停恢复按钮不应出现在纯审计页。提交输入、审核通过、驳回、交付确认等恢复动作应出现在业务运行详情或我的待办详情中；管理员取消、重试、补偿应出现在运行监控中。

## 10. 第一阶段推荐闭环

第一阶段目标先跑通一个可演示、可验证、可扩展的闭环：

```text
用户输入 -> 智能体分析/追问 -> 并行子智能体获取数据 -> 合并组装 -> 人工审核 -> 文档/邮件/OA 交付
```

第一阶段必须具备：

- 登录和当前用户 / 租户 / 角色上下文。
- 用户、部门、角色、成员和权限策略基础管理。
- 工作流草稿、固定节点类型、画布和右侧配置面板。
- 智能体模板、Skills、MCP、提示词模板和交付能力基础资产。
- 变量声明、上游引用和输出 Schema。
- 并行节点组和合并节点。
- 暂停、恢复、追问、审核、重试。
- 运行详情和审计证据链。
- PostgreSQL + Flyway 数据库版本管理。

第一阶段暂缓：

- 知识库 / RAG。
- 插件市场。
- 复杂嵌套子流程。
- 多模型自动路由。
- 完整企业 SSO / MFA。
- 高级报表分析和成本优化。
