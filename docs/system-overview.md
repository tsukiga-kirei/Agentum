# 系统详细梳理介绍

## 1. 产品定位

Agentum 是一个以智能体为载体的企业工作流平台。

它和常见 AI 聊天式工作流的关键区别是：用户不是在一个聊天框里等待 AI 一次性完成所有事，而是在一条明确的业务流程中，按步骤完成输入、确认、追问、审核、回退、暂停、恢复和交付。每个节点负责具体任务，每个智能体节点只在明确上下文里使用当前主体可用能力池中的 Skills、MCP、提示词模板和模型配置。

核心目标：

- 让企业 SOP 可以被拆成可执行、可暂停、可审计的步骤。
- 让智能体能力可复用、可分配、可版本化。
- 让并行子智能体加快数据获取和分析，再由合并节点组装结果。
- 让最终结果能落到 Word / PDF、邮件、OA、IM、Webhook 或数据库等交付目标。
- 让业务用户看懂自己正在做哪一步，而不是被复杂编排、变量和工具配置打扰。

第一阶段暂不推进知识库 / RAG 作为核心功能。后续如果业务需求变重，再作为能力资产补回，并单独处理版本、权限、引用来源和审计。

## 2. 核心概念

| 概念 | 说明 |
| --- | --- |
| 租户 | 平台最高业务隔离边界，承载租户级模型额度、可用 MCP、可用 Skills、交付能力和系统策略 |
| 组织 / 部门 | 租户内人员组织结构，用于成员管理、待办分派、审核范围和数据筛选 |
| 空间 | 租户内资源协作边界，可用于区分业务线、项目组或环境 |
| 用户 | 登录主体，一个用户可以在不同租户或空间拥有不同角色 |
| 角色 | 入口和职责集合，例如执行人、审核人、流程设计者、能力管理员、租户管理员、系统管理员。第一阶段租户管理员通过“租户管理”入口承担租户内管理职责 |
| 资源范围 | 控制当前用户、部门或角色能进入哪些业务模块、流程模板、运行记录和交付物；能力资产不拆查看 / 使用 / 执行 / 管理动作，进入可用能力池后即可用于设计与组合 |
| 智能体模板 | 可复用的智能体配置，包含系统提示词、Skills、MCP、模型、输出 Schema 和交互模式 |
| Skill | 提示词和方法论能力，例如需求拆解、追问澄清、风险识别、报告撰写 |
| MCP | 外部系统、工具或数据源连接能力，例如文件读取、数据库只读查询、邮件发送、OA 创建流程 |
| 提示词模板 | 可复用的 Prompt 片段或业务话术模板，供智能体模板或节点引用 |
| 交付能力 | 生成文件、发送邮件、创建 OA、推送 IM、Webhook、写数据库等最终动作 |
| 工作流定义 | 设计态流程模板，优先按阶段积木组织，包含步骤、依赖、变量和节点配置 |
| 工作流运行 | 某一次实际执行实例，保存状态、变量快照、节点记录、暂停事件和交付记录 |

## 3. 产品区域

Agentum 不应该让所有用户进入同一个默认编排器。不同角色看到不同入口和操作。

| 区域 | 面向角色 | 主要内容 |
| --- | --- | --- |
| 认证与账号区 | 所有人 | 登录、会话过期、无权限、个人资料、账号安全、当前租户 / 空间 / 角色 |
| 业务工作台 | 业务人员、执行人、审核人 | 我的待办、可用流程、我发起的流程、运行详情、待办处理、交付物 |
| 流程设计 | 流程设计者、智能体管理员 | 工作流列表、阶段积木编排、节点配置、变量、测试运行、发布校验 |
| 能力资产 | 能力管理员、智能体管理员 | 智能体模板、Skills、MCP、提示词模板、交付能力资产 |
| 运行监控 | 流程负责人、租户管理员 | 运行状态、失败重试、取消、补偿和管理员介入 |
| 运行审计 | 审计人员、管理员、流程负责人 | 只读运行记录、节点记录、变量快照、工具调用、审核记录、交付记录 |
| 租户管理 | 租户管理员 | 当前租户内的成员、部门、角色、空间、资源范围、需求配置和可用能力分配 |
| 系统管理 | 系统管理员 | 租户、模型供应商、全局 MCP / Skills 开关、交付能力、系统参数、多租户策略 |

前端隐藏入口只是降低干扰，不是安全边界。所有接口、资源访问、发布、执行、MCP 调用、凭证和交付动作都必须由后端重新校验。

### 3.1 登录与租户选择

登录页需要同时明确“用户要以什么入口进入”和“非系统管理员归属哪个租户”。租户是平台最高业务隔离边界，业务用户和租户管理员必须在登录时选择租户，后端据此校验用户是否属于该租户、是否具备所选入口对应角色，并把当前租户写入会话或 JWT 的活跃角色上下文。审核人、流程设计者、能力管理员等属于租户内第二层权限，不作为登录页入口角色。这里的租户管理员不是平台级系统管理员，而是当前租户内的管理入口，负责租户内组织、权限、空间和需求配置。

系统管理员入口不绑定某个租户。系统管理员登录后进入平台级系统管理，再在具体租户管理、租户可用能力池配置、租户统计或跨租户诊断场景中选择目标租户。这样可以避免把平台级操作误认为某个租户内操作。

第一阶段可参考 AuraOA 的登录链路：公开接口返回活跃租户轻量列表，登录请求携带 `tenantId` 和 `preferredRole`；后端签发令牌时写入 `activeRole`、`tenantId`、`spaceId` 和角色集合。公网 SaaS 场景下，公开租户列表可能暴露租户名称，后续可改为租户编码输入、企业域名识别或子域名识别。

### 3.2 页面抽象原则

当前前端页面是为了降低用户理解成本而抽象出的产品视图，不等同于稳定的领域模型、数据库边界或最终信息架构。

页面可以随着开发推进继续拆分、合并或改名。判断一个页面是否应该保留，优先看它是否服务某类角色的具体任务，而不是看它是否刚好对应一个后端表或一个概念名。

设计原则：

- 页面是角色任务入口，领域对象是后端事实来源。
- 页面可以聚合多个底层对象，例如运行审计页可以聚合运行记录、节点记录、等待事件、审计日志、变量快照、MCP 调用和交付记录。
- 页面不应制造新的权责边界。业务处理、运行监控和运行审计应分别对应不同操作权限。
- 过渡期为了说明概念出现的面板或卡片，如果后续被具体模块覆盖，应及时移除，避免形成重复抽象。
- 产品文案优先使用业务语言，技术动作码、表名和事件名只作为次级信息或开发调试信息。
- 管理台页面应以系统管理为母版逐层展开：先统一页头与模块切换，再进入列表、详情、表单和授权矩阵；租户管理、能力资产、审计等页面保持同一视觉节奏，避免每页自定义一套导航。
- 当前静态页面里的大卡片、说明面板和概览区主要用于表达产品心智，不是生产最终页面形态。生产页面应逐步改为“左侧大菜单 + 模块内页签 / 顶层菜单 + 列表 / 详情 / 表单 / 审计记录”的结构。
- 页面改造必须保留勾稽逻辑：系统管理配置哪些能力进入租户可用能力池，租户管理再把能力池分配给用户、部门或角色；工作流引用了哪些资产，运行审计才能追溯对应版本、输入输出、工具调用和交付记录。
- 管理台列表默认使用分页查询与分页展示，页面切换、筛选和排序都基于分页参数，不再假设一次性加载全部数据。

## 4. 业务工作台说明

业务工作台是普通用户的默认入口，不默认展示设计态编排。

### 4.1 我的待办

“我的待办”表示流程已经暂停，并且下一步需要当前用户、当前角色或当前部门处理。

典型待办：

- 用户输入节点等待发起人补充资料。
- 智能体追问确认等待用户回答或确认。
- 人工审核节点等待审核人通过、驳回或修改后继续。
- 高风险 MCP 调用等待审批。
- 交付节点等待交付确认。

待办的核心问题是：现在轮到谁做什么。

### 4.2 运行态摘要

“运行态摘要”表示用户有权限查看的流程运行位置和状态，不一定需要用户处理。

典型状态：

- 运行中：节点正在执行。
- 已暂停：等待某人输入、审核、审批或外部回调。
- 已完成：可以查看交付物。
- 已失败：可以查看错误和是否允许重试。

运行态摘要的核心问题是：流程现在在哪里，为什么。

### 4.3 可用流程模板

可用流程模板是业务用户可以发起的已发布流程。点击后不是进入设计态编排，而是创建一次工作流运行，并进入业务态交互页面。

业务态交互页面应支持：

- 按步骤填写资料。
- 查看智能体已完成的摘要。
- 对智能体结果追问或要求改写。
- 确认后进入下一步。
- 流程暂停、退出后恢复。
- 有权限时回退到上一步或取消本次运行。
- 查看最终交付物。

### 4.4 业务运行详情

业务运行详情承接“我的待办”和“运行态摘要”的下一层页面。它面向发起人、当前待办处理人和流程负责人，不面向纯审计人员。

业务运行详情可以出现处理动作：

- 提交用户输入。
- 回答智能体追问。
- 确认或改写智能体输出。
- 人工审核通过、驳回或修改后继续。
- 审批高风险 MCP 调用。
- 确认交付。

这些动作本质上都是“恢复暂停点”，后端必须记录恢复事件、操作人、意见、附件和来源渠道，并重新校验权限。

## 5. 流程设计积木

流程设计不以自由画布作为主要体验。Agentum 的工作流设计应像“搭积木”：左侧展示可排序的步骤积木，右侧展示当前积木的具体配置。设计者关注的是每一步谁输入、哪个智能体处理、调用哪些 Skill / MCP、输入输出参数如何衔接，以及最终如何交付。

流程设计页签第一阶段按三块组织：

| 页签 | 目标 | 典型内容 |
| --- | --- | --- |
| 总览 | 说明流程设计策略和积木类型 | 积木说明、最近工作流、发布边界 |
| 工作流定义 | 管理草稿并进入具体配置 | 工作流定义卡片、搜索、新建、进入积木配置 |
| 发布校验 | 发布前检查并冻结版本 | 校验步骤结构、变量依赖、能力引用和交付配置 |

当前可选积木先收敛为四类：

| 积木 | 作用 | 右侧主要配置 |
| --- | --- | --- |
| 输入节点 | 收集用户输入信息 | 输入框名称、输出参数名、占位提示 |
| 单智能体节点 | 运行一个智能体完成分析、生成或追问 | 选择智能体或自定义提示词，选择提示词模板、MCP、Skill，配置输入输出参数和交互模式 |
| 智能体集群节点 | 组合多个单智能体并行或协作 | 子智能体列表、每个智能体提示词、输出参数、拼接 / 汇总规则 |
| 交付节点 | 生成并交付最终结果 | 交付方式、交付内容变量、交付说明 |

授信报告这类流程的推荐链路：

```text
输入授信公司全称
  -> 智能体识别缺失信息并追问
  -> 并行调用 MCP 获取工商、司法、财务和行业数据
  -> 多个章节智能体并行生成报告章节
  -> 报告组装智能体统一口径和格式
  -> 用户审查确认
  -> 生成并交付正式文档
```

交互规则必须区分“模型文本”和“事实数据”：

- 模型生成的分析、章节、措辞和摘要可以支持重新生成、追问修改或人工改写后继续。
- MCP 查询、数据库读取、司法记录、财务指标等事实类数据不能让用户通过追问直接改写；只能重新获取、补充来源或进入人工审核说明。
- 每个积木都要展示当前任务、输入参数、输出参数、可执行动作和下一步去向，弥补开源模型在长任务规划、状态保持和工具约束上的不足。
- 智能体集群节点应优先服务“数据获取并行”和“章节生成并行”，并在积木内配置拼接 / 汇总规则后输出下游统一变量。

工作流由固定节点类型组成。固定节点降低复杂度，能力复用来自节点引用的资产。

| 节点 | 作用 | 是否可暂停 |
| --- | --- | --- |
| 触发节点 | 手动、定时、Webhook 或外部事件触发流程 | 否 |
| 用户输入节点 | 表单、文件上传、人员选择、部门选择、补充说明 | 是 |
| 智能体节点 | 调用智能体模板执行分析、生成、判断、提取或追问 | 可选 |
| 并行节点组 | 同时启动多个子智能体或工具任务，加快数据获取 | 取决于子节点 |
| 合并 / 组装节点 | 汇聚并行结果，透传、JSON 合并、模板组装或调用组装智能体 | 可选 |
| 条件分支节点 | 基于变量、规则或外部返回值决定路由 | 否 |
| 人工审核节点 | 人工确认、修改、通过、驳回、回退 | 是 |
| 交付节点 | 生成文件、发送邮件、创建 OA、推送 IM、Webhook、写数据库 | 否 |
| 结束节点 | 结束本次运行并固化结果 | 否 |

并行节点的本意是放入多个子智能体或工具任务并发执行，例如“资料核验智能体、外部数据查询智能体、风险补充分析智能体”同时运行。合并节点负责把它们的输出组装为下游可用变量，必要时可以绑定“报告组装智能体”。

输入节点、条件节点、结束节点这类不依赖智能体的通用节点，建议平台内置。是否也作为“能力资产”管理，后续可根据复用和版本治理需要再决定；第一阶段先按内置节点处理更简单。

## 6. 能力资产

能力资产是流程设计的复用来源。流程节点不复制资产内容，而是引用资产版本。

第一阶段能力资产范围：

- 智能体模板。
- Skills。
- MCP。
- 提示词模板。
- 交付能力资产。

暂不展示知识库资产。

能力资产分为两层：

- 源码 / 实现层：放在仓库 `capabilities/` 目录下，保存产品运行时 Skills、自研 MCP Server、提示词模板源码和交付适配实现。
- 资产 / 分配层：放在数据库和管理页面中，保存资产名称、版本、发布状态、引用关系、租户可用能力池、租户内分配范围和运行安全策略。

这两层不能混为一个概念。`capabilities/skills/` 和 `capabilities/mcp-servers/` 负责让能力能被开发、测试和部署；能力资产页负责让这些能力被登记、发布、停用、版本化和追溯。

`.codex/skills/` 是开发本项目时给 Codex 使用的本地辅助技能，不是 Agentum 产品运行时的 Skill 资产来源。

能力分配采用“可用能力池”模型，不再拆成“查看、使用、执行、管理”等动作权限。系统管理员把平台能力放入某个租户的可用能力池；租户管理员再把这批能力分配给用户、部门或角色。分配完成后，用户在能力资产页、智能体模板和流程设计中自然可见并可用，不需要再理解底层动作。

“我的能力”需要区分草稿和正式能力。用户新建时只生成“能力草稿”，草稿归当前创建人维护；完善后发布为正式能力，流程设计再引用正式版本。用户可以删除自己创建的能力，但后续接入流程引用关系后，已被草稿、发布流程或运行快照引用的能力必须禁止删除。第一阶段从安全边界出发，业务用户只允许创建两类租户内资产：提示词模板草稿和智能体模板草稿。Skill、MCP 和交付能力涉及执行代码、外部连接、凭证或最终写入动作，暂不开放给业务用户自建，只能由系统管理登记、测试并放入租户可用能力池，再由租户管理分配。

提示词模板草稿编辑时保存提示词正文、适用场景和版本信息。智能体模板草稿编辑时保存系统提示词，并且只能从当前主体已开放的 Skill 和 MCP 中选择构建材料；后端发布前必须重新校验这些引用仍在当前主体能力池内，避免用户通过手写 ID 绕过租户分配和运行安全边界。

能力资产页面应聚焦“资产是什么、版本是什么、能被哪些流程引用”。租户可用能力池、人员 / 部门 / 角色分配和凭证策略不建议在这里作为主要展示内容，应放到系统管理和租户管理中，以免能力资产页变成安全配置页。

能力资产页不建议展示“流程导入建议”这类设计指导。流程节点如何引用资产，应放在流程设计的节点配置面板中。能力资产页应提供新增、发布、停用、版本管理和查看引用关系等入口。

如果每类资产模块已经提供独立新增入口，就不应再额外保留一个泛化的“资产创建入口”。右侧或辅助区域更适合展示待治理资产、引用关系、缺失授权、草稿待发布和高风险能力缺少审批策略等运营信息。

### 6.1 交付能力资产

交付能力也应成为可治理资产。

示例：

| 交付能力 | 说明 | 风险 |
| --- | --- | --- |
| Word / PDF 生成 | 根据模板和变量生成正式文档 | 中 |
| 邮件发送 | 发送正文和附件，可接 Mailpit 本地验证 | 高 |
| OA 流程创建 | 创建审批或归档流程 | 高 |
| IM 推送 | 推送飞书、钉钉、企微消息 | 中 |
| Webhook 回调 | 回调外部系统 | 中 |
| 数据库写入 | 写入业务表或结果表 | 高 |

高风险交付能力必须支持审批、二次确认、审计和失败重试。

## 7. 系统管理与租户配置

系统管理是平台级入口，负责“哪些底层能力可以被登记、测试，并在具体租户中启用”。

推荐链路：

```text
系统管理员注册并测试底层能力
  -> 在某个租户的配置抽屉中放入租户可用能力池
  -> 租户管理员在当前租户内分配给用户、部门或角色
  -> 用户在能力资产、智能体模板和流程设计中自然可见并可用
  -> 流程发布后，业务用户直接使用流程，不需要理解底层能力分配
```

能力分配只回答“这项能力是否进入当前主体的可用范围”。高风险 MCP、外部交付、数据库写入、敏感文件下载等仍需要运行安全策略，例如审批、二次确认、额度、凭证注入、脱敏和审计；这些策略不属于能力分配动作矩阵。

系统管理负责：

- 租户管理。
- 模型供应商和模型供应商类型。
- 全局 MCP 注册、测试、启停。
- 全局 Skills 和提示词模板上架。
- 全局交付能力注册。
- 在单租户抽屉中维护“租户可用能力池 / 模型分配”。
- 凭证策略和安全策略。
- 多租户隔离、资源配额、数据保留策略。

租户管理负责：

- 用户、部门、角色和成员。
- 当前租户可用能力池向用户、部门或角色的二次分配。
- 哪些用户、部门或角色可以进入业务工作台、流程设计、能力资产、运行审计等模块和页签。
- 哪些流程模板、智能体模板、运行记录和交付物进入当前主体的可见范围。
- 高风险能力在运行时是否需要审批、二次确认、额度限制或审计加严。

第一阶段把“租户内管理”明确命名为租户管理。也就是说，系统管理员负责平台级租户创建、启停、全局能力登记测试，并在系统管理的单租户抽屉中维护租户可用能力池和模型分配；租户管理员负责当前租户内的组织、空间、模块入口、需求配置和能力池分配。

系统管理生产页面建议在左侧进入“系统管理”大模块后，再按模块内页签或顶层菜单组织：

- 概览：用少量卡片说明平台租户数、启用能力、模型供应商、异常授权和待处理风险，只作为进入具体管理模块前的摘要。
- 租户管理：租户创建、启停、基础信息、租户状态、配额和数据保留策略。支持在抽屉中配置租户可用能力池、模型分配、成员角色诊断和默认信息。
- 能力管理：全局 MCP、Skills、提示词模板、交付能力的注册、测试、启停和版本治理；模型不作为全局能力登记。
- 模型管理：供应商、模型、额度、默认模型和密钥脱敏状态。

租户管理生产页面建议在左侧进入“租户管理”大模块后，再按模块内页签或顶层菜单组织：

- 概览：展示当前租户成员数、部门数、角色数、待配置入口、能力池缺口和近期风险操作。
- 人员组织：用户、部门、成员关系、空间归属和账号状态。
- 角色与入口：业务角色、模块入口、页签范围和审计可见性。
- 能力分配：把租户可用能力池分配给用户、部门或角色，分配后前端可见即能用。
- 资源范围：工作流、智能体模板、运行记录、交付物面向哪些用户、部门或角色可见。
- 需求配置：租户内流程需求、表单字段、审核规则、交付目标和可用能力约束。
- 租户内审计：当前租户内成员变更、权限变更、能力分配和敏感操作记录。

AuraOA 必须作为权限设计的重要参照：参考它的系统管理员 / 租户管理员路由分层、JWT + 租户上下文 + 角色校验、部门 / 角色 / 成员管理、租户模型和系统配置管理。Agentum 不照搬 Go 技术栈和 OA 业务字段，但保留其“平台级系统管理”和“租户内管理”分层思想；在 Agentum 中，租户内管理入口命名为“租户管理”。

## 8. 租户管理中的权限管理

权限模型采用：

```text
认证身份 + 租户上下文 + 组织 / 部门 / 空间归属 + RBAC + 模块入口 + 资源范围 + 能力池分配 + 运行安全策略 + 审计
```

推荐角色：

| 角色 | 说明 |
| --- | --- |
| 系统管理员 | 管理全局配置、租户、模型、全局 MCP、交付能力 |
| 租户管理员 | 管理当前租户内用户、部门、角色、空间、需求配置、资源范围和可用能力分配 |
| 流程设计者 | 创建和编辑工作流、配置节点、测试运行 |
| 智能体管理员 | 创建、维护和发布智能体模板 |
| 能力管理员 | 管理 Skills、MCP、提示词模板和交付能力资产 |
| 审核人 | 处理人工审核节点和高风险审批 |
| 执行人 | 发起流程、填写输入、追问确认、查看自己的结果 |
| 观察者 | 只读查看被分配范围内的流程、资产或运行记录 |

能力分配不设计“查看 / 使用 / 执行 / 管理”动作矩阵。系统管理只决定能力是否进入租户可用能力池；租户管理只决定这批能力分配给哪些用户、部门或角色。分配完成后，前端用户在能力资产、智能体模板和流程设计中看到的就是可用材料。

流程、运行记录、交付物和审计仍然需要模块入口和资源范围控制。例如谁可以进入流程设计、谁可以发布流程、谁可以处理审核、谁可以查看审计证据链，这些属于业务模块权限，不应混入能力资产授权动作。

敏感能力的审批、二次确认、额度、凭证注入、脱敏和审计属于运行安全策略。它们用于保护高风险 MCP、外部交付、数据库写入、敏感文件下载等调用，不改变“能力分配后可见即能用”的产品心智。

第一阶段以三类登录入口作为第一重权限：

- 业务用户：进入业务侧工作区，可承载业务工作台、流程设计、能力资产、运行审计等大模块；实际能看到哪些模块、页签、流程和资产，由租户管理里的第二重配置决定。
- 租户管理员：进入租户管理，管理当前租户内成员、部门、角色、菜单、资源范围、需求配置和能力池分配。
- 系统管理员：进入系统管理，管理平台租户、全局能力登记测试、模型供应商、租户可用能力池 / 模型分配和凭证策略。

一个用户可以同时拥有多个角色或多个租户身份，必须设计角色 / 租户切换。切换时应参照 AuraOA 的做法重新计算活跃租户、活跃角色、左侧菜单、模块内页签、资源范围和可用能力池；前端只做展示切换，后端必须在每次接口调用时按 token、租户上下文、角色、部门、人员分配和资源范围重新校验。

菜单权限分两层：

- 第一层是登录入口角色：`business`、`tenant_admin`、`system_admin`。它决定用户进入业务侧、租户管理还是系统管理，不等同于租户内业务职务。
- 第二层是租户内配置：租户管理员按用户、部门、租户自定义角色分配业务工作台、流程设计、能力资产、运行审计等大模块、模块内页签、资源范围和可用能力池。用户没有被分配对应模块、页签或资源时，前端不展示，后端仍按资源范围复核。
- 模块内页签 / 顶层菜单表示当前大模块内部的详细功能区，例如租户管理里的人员组织、角色与入口、能力分配、资源范围、运行安全策略、租户内审计。它们不是全局顶层菜单，也不应全部做成首页卡片入口。

租户管理员负责当前租户内的入口、资源范围和能力池分配，可以按角色、部门、人员三个维度配置。人员级分配用于补充特殊职责，必须有原因、有效期和审计记录，避免绕过角色 / 部门策略后不可追溯。

租户管理的治理配置建议按页签组织：

- 人员组织：用户、部门、空间成员关系。
- 角色与入口：角色能进入哪些业务模块和模块内页签。
- 能力分配：租户可用能力池分配给哪些用户、部门或角色，分配后可见即能用。
- 资源范围：具体流程、智能体、运行记录、交付物面向哪些用户、部门或角色可见。
- 运行安全策略：高风险 MCP、外部交付、敏感文件下载、凭证申请和审批。
- 审计可见性：谁可以查看只读证据链，以及哪些字段需要脱敏。

凭证配置和凭证明文读取不属于租户管理的主要职责。系统级凭证策略、模型密钥和全局 MCP 凭证放在系统管理；租户管理只决定当前租户内能力池如何分配，以及高风险调用是否需要申请、审批或审计加严。

## 9. 运行态交互

运行态是 Agentum 的核心体验，应围绕“当前步骤”而不是“聊天历史”组织。

必须支持：

- 暂停：等待用户输入、追问、人工审核、外部回调或高风险审批。
- 恢复：提交表单、确认结果、审核通过、审批工具调用后继续。
- 追问：用户围绕当前智能体结果继续问，确认后写入最终变量。
- 改写：用户要求智能体重写当前节点输出，但仍受输出 Schema 约束。
- 回退：有权限时退回到上一个可回退节点，并保留审计记录。
- 退出：用户离开页面后可从待办或运行记录恢复。
- 重试：节点失败后按策略重试，并记录重试原因和结果。

运行详情应展示：

- 当前状态。
- 当前节点。
- 等待对象。
- 暂停原因。
- 输入变量和输出变量。
- 已完成步骤摘要。
- 工具调用摘要。
- 审核记录。
- 交付记录。
- 最终交付物。

业务用户看到简化步骤条，设计者看到阶段积木配置，审计人员进入只读证据链。

### 9.1 审核、监控与审计边界

“审核”和“审计”必须分开：

- 审核是业务流程中的人工节点，由审核人或流程负责人决定通过、驳回、修改后继续。
- 运行监控是管理员视角，用于查看运行状态、处理失败重试、取消和补偿，所有介入动作都必须写入审计。
- 运行审计是只读证据链，用于还原流程当时发生了什么，不推动业务流程继续。

运行审计可以展示执行链路、节点输入输出快照、变量快照、MCP 调用摘要、模型输出解析、审核记录和交付记录。它们的意义是追溯“结论从哪里来、谁在什么时候做了什么、敏感数据是否脱敏、工具调用是否越权”，不是让审计人员修改当前流程。

“执行链路”是聚合视图，不是新的独立数据源。它应由 `WorkflowRun`、`NodeRun`、`WaitingEvent`、`RunEvent` / `audit_logs`、`mcp_call_logs`、`delivery_records` 和 `variable_snapshots` 组合生成。审计日志是证据来源之一，但不应被设计成承载所有执行状态的万能流水表。

暂停恢复按钮不应出现在纯审计页。提交输入、审核通过、驳回、交付确认等恢复动作应出现在业务运行详情或我的待办详情中；管理员取消、重试、补偿应出现在运行监控中。

## 10. 第一阶段推荐闭环

第一阶段目标先跑通一个可演示、可验证、可扩展的闭环：

```text
用户输入 -> 智能体分析/追问 -> 并行子智能体获取数据 -> 合并组装 -> 人工审核 -> 文档/邮件/OA 交付
```

第一阶段必须具备：

- 登录和当前用户 / 租户 / 角色上下文。
- 用户、部门、角色、成员和权限策略基础管理。
- 工作流草稿、固定节点类型、阶段积木和右侧配置面板。
- 智能体模板、Skills、MCP、提示词模板和交付能力基础资产。
- 变量声明、上游引用和输出 Schema。
- 并行节点组和合并节点。
- 暂停、恢复、追问、审核、重试。
- 运行详情和审计证据链。
- PostgreSQL + Flyway 数据库版本管理。

第一阶段暂缓：

- 知识库 / RAG。
- 插件市场。
- 复杂嵌套子流程。
- 多模型自动路由。
- 完整企业 SSO / MFA。
- 高级报表分析和成本优化。
