# Agentum 系统架构

## 1. 架构目标

Agentum 的架构设计目标是长期可维护，而不是只追求第一版开发速度。

核心要求：

- 领域边界清晰，避免所有逻辑堆在画布或执行器里。
- 工作流定义和运行实例分离，保证发布版本可回放。
- 智能体、Skills、MCP、知识库、模型配置都作为资产独立管理。
- 所有外部调用经过统一网关，方便权限、审计、限流和密钥隔离。
- 用户输入、追问、人工审核、Webhook 回调统一建模为暂停与恢复。

## 2. 推荐整体架构

```text
Web 前端
  |
  | OpenAPI / REST
  v
API 服务
  |
  |-- 认证与权限模块
  |-- 工作流设计模块
  |-- 工作流执行模块
  |-- 智能体资产模块
  |-- Skills 资产模块
  |-- MCP 网关模块
  |-- 审计日志模块
  |-- 交付模块
  |
  v
PostgreSQL / Redis / 对象存储 / 消息队列
  |
  v
Worker 集群
  |
  |-- 模型调用
  |-- MCP 调用
  |-- 文档生成
  |-- 文件解析
  |-- 外部系统推送
```

## 3. 模块边界

### 3.1 Web 前端

职责：

- 工作流画布展示与编辑。
- 节点配置表单。
- 资产库管理页面。
- 工作流运行态页面。
- 人工输入、追问、审核操作。
- 执行日志与交付物查看。

前端不负责：

- 权限最终判断。
- MCP 凭证管理。
- 模型调用。
- 工作流真实执行。

### 3.2 API 服务

职责：

- 对外提供统一 API。
- 做认证、授权和参数校验。
- 保存工作流定义、资产配置和运行记录。
- 创建工作流运行实例。
- 分发异步任务给 Worker。
- 提供审计日志查询。

### 3.3 工作流执行模块

职责：

- 读取已发布的工作流定义。
- 按节点和边生成执行计划。
- 管理节点运行状态。
- 处理变量注入、条件判断、并行合并。
- 处理暂停和恢复。
- 写入运行日志和变量快照。

该模块是 Agentum 的核心，必须保持独立，不应直接依赖前端展示逻辑。

### 3.4 智能体运行模块

职责：

- 组装系统提示词、用户输入、变量上下文、Skills 和 MCP 工具声明。
- 调用模型 Provider。
- 解析结构化输出。
- 处理追问、多轮确认和一次性输出。
- 将结果写回变量系统。

### 3.5 MCP 网关模块

职责：

- 注册 MCP 服务。
- 管理 MCP 凭证。
- 校验用户和智能体是否有调用权限。
- 记录 MCP 调用参数、耗时、结果摘要和错误。
- 对敏感工具做二次确认或审批。

### 3.6 交付模块

职责：

- 根据变量生成 Word / PDF。
- 发送邮件或 IM。
- 创建 OA 流程。
- 写入数据库。
- 调用外部 Webhook。
- 记录交付结果。

## 4. 关键设计边界

### 4.1 定义与实例分离

工作流定义不可直接代表一次运行。

- `WorkflowDefinition`：描述流程长什么样。
- `WorkflowVersion`：某次发布后的不可变版本。
- `WorkflowRun`：一次实际运行。
- `NodeRun`：某个节点在某次运行中的状态和产物。

### 4.2 资产引用而不是复制

工作流节点引用智能体、Skill、MCP 时，应记录引用版本。

示例：

```json
{
  "agentTemplateId": "agt_requirement_analysis",
  "agentTemplateVersion": 3
}
```

这样可以保证历史运行可追溯，同时允许资产继续演进。

### 4.3 执行器不关心 UI

执行器只理解节点协议、变量、状态和事件，不理解画布坐标、颜色、折叠状态等 UI 信息。

画布信息应单独放在 `uiSchema` 或 `layout` 字段里。

## 5. 推荐服务拆分顺序

第一阶段可以做单体后端，但内部模块边界要清晰。

推荐顺序：

1. 单体服务：API、工作流、资产、权限、审计都在一个 Spring Boot 应用中。
2. Worker 独立：将模型调用、文件处理、交付任务拆成后台 Worker。
3. MCP 网关独立：当外部系统连接变多后单独拆分。
4. 执行引擎独立：当流程运行量和调度复杂度提升后再拆。

不要第一版就拆太多服务，否则会增加部署和排错成本。
