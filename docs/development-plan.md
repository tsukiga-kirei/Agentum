# Agentum 开发计划

本文档用于把产品、架构和工程规范落成可执行任务。每次开发前应先阅读 `AGENTS.md`，再按本计划选择当前阶段任务。

## 1. 当前阶段

当前处于 **阶段一：核心闭环**。

阶段一目标是跑通一个可演示、可验证、可继续扩展的最小流程：

```text
用户输入 -> 智能体分析 -> 用户确认/追问 -> 并行数据获取 -> 合并组装 -> 人工审核 -> 文档/邮件交付
```

阶段一不追求完整企业治理能力，优先让工作流定义、配置、运行、暂停、恢复、审计和交付形成闭环。

## 2. 开发顺序

### 2.1 基础应用壳

目标：

- 建立前端业务工作台基础布局。
- 区分业务使用区、流程设计工作台、能力管理区、系统管理区和运行审计区。
- 提供稳定导航、页面标题、状态摘要和主要入口。

验收标准：

- 用户进入前端后能看到工作台、待办、流程、资产、审计等核心入口。
- 布局符合 `docs/frontend-guidelines.md`，不是营销页。
- 页面在桌面宽度下信息层级清晰，移动端至少可浏览。

### 2.2 共享契约与类型生成

目标：

- 基于 `packages/shared-contract` 中的 OpenAPI 和 JSON Schema 固定第一批前后端契约。
- 补充工作流、节点、变量、运行记录、智能体模板和审核动作的类型定义。
- 建立前端类型引用方式，避免手写重复类型。

验收标准：

- 工作流定义、节点配置和运行状态有统一类型来源。
- 契约变更能被前后端构建或测试及时发现。
- 文档中说明类型生成和校验命令。

### 2.3 工作流定义管理

目标：

- 后端提供工作流创建、编辑、保存、查询和版本草稿能力。
- 前端提供工作流列表、基础信息编辑和节点配置入口。
- 先支持草稿保存，不急于做复杂发布策略。

验收标准：

- 可以创建一个工作流草稿。
- 可以保存固定节点类型和节点间关系。
- API 返回结构符合 `docs/development-standards.md`。

### 2.4 画布与节点配置

目标：

- 使用 React Flow 搭建工作流编辑器。
- 支持固定节点类型：触发、用户输入、智能体、并行、合并、条件、人工审核、交付。
- 使用右侧配置面板编辑节点基础信息、输入输出变量和核心配置。

验收标准：

- 节点颜色、尺寸、选中态和配置面板符合前端规范。
- 节点保存前有基础校验。
- 智能体节点、用户输入节点、人工审核节点和交付节点至少具备 MVP 配置字段。

### 2.5 变量系统

目标：

- 建立变量声明、引用和下游传递规则。
- 支持用户输入、智能体输出、人工审核结果和交付结果写入变量。
- 前端提供变量选择和插入能力。

验收标准：

- 节点只能引用上游可见变量。
- 变量类型和必填规则能被校验。
- 变量解析错误返回明确错误码和中文 message。

### 2.6 智能体与能力资产

目标：

- 实现智能体模板基础管理。
- 支持为智能体节点绑定模型、系统提示词、Skills、MCP 和知识库占位配置。
- 第一阶段可先模拟模型调用结果，再接真实模型供应商。

验收标准：

- 智能体节点配置能保存并回显。
- 输出 schema 能被校验。
- 模型调用失败、输出格式错误和权限不足有清晰错误码。

### 2.7 运行引擎与暂停恢复

目标：

- 后端实现最小工作流运行状态机。
- 支持用户输入、智能体节点、并行节点组、合并、条件、人工审核和交付节点的基础执行。
- 支持流程暂停、恢复、失败和重试。

验收标准：

- 可以启动一次工作流运行。
- 流程能在用户输入或人工审核节点暂停。
- 恢复后能继续执行下游节点。
- 核心状态流转有单元测试。

### 2.8 运行详情与审计证据链

目标：

- 前端提供运行详情页。
- 展示当前状态、当前节点、暂停原因、执行日志、变量快照、工具调用、用户输入、审核记录和交付记录。
- 后端记录每个节点的输入、输出、耗时、异常、重试和人工操作。

验收标准：

- 用户能清楚知道流程为什么停住、下一步谁处理。
- 审计日志不可缺少关键节点事件。
- 运行详情符合 `docs/frontend-workflow-visualization.md`。

### 2.9 基础交付能力

目标：

- 第一阶段至少实现一种交付方式，优先选择邮件调试或文档生成。
- 交付节点能读取变量并生成交付记录。
- 失败时支持错误展示和重试。

验收标准：

- 成功运行的流程能产生一个可查看的交付结果。
- 交付失败有明确错误码和重试记录。
- 本地开发可通过 Mailpit 或文件产物验证结果。

## 3. 阶段一完成标准

阶段一完成时必须满足：

- 一个 MVP 流程可以从创建、配置、启动、暂停、恢复、审核到交付完整跑通。
- 前端至少包含工作台、工作流列表、编辑器、运行详情和基础资产入口。
- 后端至少包含工作流定义、运行状态机、变量系统、审计日志和基础交付 API。
- 核心逻辑有测试覆盖，尤其是状态机、变量解析、权限判断、输出校验和重试。
- 本地开发命令和必要环境变量在文档中可复现。

## 4. 后续阶段

### 阶段二：资产治理

- Skills 库
- MCP 服务库
- 智能体模板库
- 版本管理
- 权限管理
- 审计日志增强

### 阶段三：企业增强

- 多租户
- SSO
- 高级审批
- 生产/测试环境隔离
- 更细粒度的 MCP 凭证管理
- 流程运行分析

### 阶段四：生态扩展

- 插件市场
- 模板市场
- 第三方系统连接器
- 私有化部署方案
- 多模型策略与成本优化

## 5. 每次任务的交付要求

- 开始前确认所属阶段和任务编号。
- 修改范围应尽量贴近当前任务，不顺手做无关重构。
- 新增或修改接口时同步契约、文档和测试。
- 新增或修改复杂逻辑时同步补充中文注释；静态数据、模拟交互和后续要接 API 的位置必须标明替换方向。
- 完成后运行对应验证，并在最终说明中写明验证结果。
- 若发现计划与实现冲突，先更新本文档，再继续开发。
