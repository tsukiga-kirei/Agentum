# Agentum 开发计划

本文档用于把产品、架构和工程规范落成可执行任务。每次开发前应先阅读 `AGENTS.md`，再按本计划选择当前阶段任务。

## 1. 当前阶段

当前处于 **阶段一：核心闭环**。

阶段一目标是跑通一个可演示、可验证、可继续扩展的最小流程：

```text
用户输入 -> 智能体分析 -> 用户确认/追问 -> 并行数据获取 -> 合并组装 -> 人工审核 -> 文档/邮件交付
```

阶段一不追求完整企业治理能力，但必须先具备最小认证、用户、组织、角色和空间权限基础。否则后续工作流、资产、审计和交付都无法判断“谁在操作、属于哪个组织、以什么角色访问哪些资源”。

阶段一的顺序调整为：

```text
认证与组织权限基础 -> 工作台壳层 -> 工作流定义与画布 -> 变量与能力资产 -> 运行暂停恢复 -> 审计与交付
```

## 2. 开发顺序

### 2.0 认证、用户与组织权限基础

前端进度：待补齐。当前仅有权限策略占位页，还没有登录页、个人资料页、用户管理、组织管理、角色管理和空间切换。

目标：

- 提供企业后台登录体验，支持账号密码登录、登录错误提示、会话过期提示和 SSO / MFA 占位入口。
- 提供用户个人资料页，展示头像、姓名、邮箱、手机号、所属组织、当前角色、主题偏好和安全设置。
- 提供用户管理页，支持用户列表、状态、所属组织、所属空间、角色分配和禁用提示。
- 提供组织与空间管理页，支持组织、空间、成员、邀请、默认空间和空间管理员设置。
- 提供角色管理页，支持系统角色、空间角色、业务角色、审核角色和执行角色的权限矩阵。
- 让前端导航、页面入口和操作按钮基于当前用户角色做显示控制，但最终权限仍由后端校验。

验收标准：

- 用户未登录时只能看到登录页，不能进入工作台、设计器、资产、审计或系统管理区。
- 登录成功后能看到当前用户、当前组织、当前空间和当前角色。
- 用户可以查看和编辑个人资料中的非敏感字段，密码、MFA 和密钥类操作必须走安全确认。
- 空间管理员可以查看成员列表、角色分配和邀请状态。
- 系统管理员可以查看组织、用户、角色和权限策略入口。
- 前端权限隐藏只作为体验优化，所有敏感操作文案必须提示“后端会重新校验权限”。

### 2.1 基础应用壳

前端进度：已完成。当前已提供企业工作台壳层、分区导航、深浅主题、侧栏折叠、状态摘要和核心入口。

目标：

- 建立前端业务工作台基础布局。
- 区分业务使用区、流程设计工作台、能力管理区、系统管理区和运行审计区。
- 提供稳定导航、页面标题、状态摘要和主要入口。

验收标准：

- 用户进入前端后能看到工作台、待办、流程、资产、审计等核心入口。
- 布局符合 `docs/frontend-guidelines.md`，不是营销页。
- 页面在桌面宽度下信息层级清晰，移动端至少可浏览。

### 2.2 共享契约与类型生成

前端进度：前端占位完成。当前补充了首批 OpenAPI / JSON Schema 字段，并在 `apps/web/src/types/workflow-contract.ts` 中提供临时类型出口；后续应替换为生成产物。

目标：

- 基于 `packages/shared-contract` 中的 OpenAPI 和 JSON Schema 固定第一批前后端契约。
- 补充工作流、节点、变量、运行记录、智能体模板和审核动作的类型定义。
- 建立前端类型引用方式，避免手写重复类型。

验收标准：

- 工作流定义、节点配置和运行状态有统一类型来源。
- 契约变更能被前后端构建或测试及时发现。
- 文档中说明类型生成和校验命令。

### 2.3 工作流定义管理

前端进度：前端可演示完成。当前支持工作流列表、搜索、新建草稿、发布校验入口和打开画布；数据仍为内存态模拟。

目标：

- 后端提供工作流创建、编辑、保存、查询和版本草稿能力。
- 前端提供工作流列表、基础信息编辑和节点配置入口。
- 先支持草稿保存，不急于做复杂发布策略。

验收标准：

- 可以创建一个工作流草稿。
- 可以保存固定节点类型和节点间关系。
- API 返回结构符合 `docs/development-standards.md`。

### 2.4 画布与节点配置

前端进度：前端可演示完成。当前支持固定节点类型画布、节点卡片、节点搜索定位、左右面板收起、右侧配置分组和节点保存模拟。

目标：

- 使用 React Flow 搭建工作流编辑器。
- 支持固定节点类型：触发、用户输入、智能体、并行、合并、条件、人工审核、交付。
- 使用右侧配置面板编辑节点基础信息、输入输出变量和核心配置。

验收标准：

- 节点颜色、尺寸、选中态和配置面板符合前端规范。
- 节点保存前有基础校验。
- 智能体节点、用户输入节点、人工审核节点和交付节点至少具备 MVP 配置字段。

### 2.5 变量系统

前端进度：前端可演示完成。当前支持上游变量可见性过滤、变量面板、敏感变量提示和点击插入变量占位。

目标：

- 建立变量声明、引用和下游传递规则。
- 支持用户输入、智能体输出、人工审核结果和交付结果写入变量。
- 前端提供变量选择和插入能力。

验收标准：

- 节点只能引用上游可见变量。
- 变量类型和必填规则能被校验。
- 变量解析错误返回明确错误码和中文 message。

### 2.6 智能体与能力资产

前端进度：前端可演示完成。当前能力资产页展示智能体模板、Skills、MCP、知识库、版本、授权角色和审计要求。

目标：

- 实现智能体模板基础管理。
- 支持为智能体节点绑定模型、系统提示词、Skills、MCP 和知识库占位配置。
- 第一阶段可先模拟模型调用结果，再接真实模型供应商。

验收标准：

- 智能体节点配置能保存并回显。
- 输出 schema 能被校验。
- 模型调用失败、输出格式错误和权限不足有清晰错误码。

### 2.7 运行引擎与暂停恢复

前端进度：前端演示占位完成。当前运行审计页展示运行状态、暂停原因、等待对象和恢复操作按钮；真实状态流转仍依赖后端。

目标：

- 后端实现最小工作流运行状态机。
- 支持用户输入、智能体节点、并行节点组、合并、条件、人工审核和交付节点的基础执行。
- 支持流程暂停、恢复、失败和重试。

验收标准：

- 可以启动一次工作流运行。
- 流程能在用户输入或人工审核节点暂停。
- 恢复后能继续执行下游节点。
- 核心状态流转有单元测试。

### 2.8 运行详情与审计证据链

前端进度：前端可演示完成。当前提供运行步骤、变量快照、审计日志、MCP 调用摘要和敏感变量脱敏提示。

目标：

- 前端提供运行详情页。
- 展示当前状态、当前节点、暂停原因、执行日志、变量快照、工具调用、用户输入、审核记录和交付记录。
- 后端记录每个节点的输入、输出、耗时、异常、重试和人工操作。

验收标准：

- 用户能清楚知道流程为什么停住、下一步谁处理。
- 审计日志不可缺少关键节点事件。
- 运行详情符合 `docs/frontend-workflow-visualization.md`。

### 2.9 基础交付能力

前端进度：前端演示占位完成。当前运行审计页展示 Word / PDF 与邮件交付记录、待交付状态和失败可重试提示。

目标：

- 第一阶段至少实现一种交付方式，优先选择邮件调试或文档生成。
- 交付节点能读取变量并生成交付记录。
- 失败时支持错误展示和重试。

验收标准：

- 成功运行的流程能产生一个可查看的交付结果。
- 交付失败有明确错误码和重试记录。
- 本地开发可通过 Mailpit 或文件产物验证结果。

## 3. 阶段一完成标准

阶段一完成时必须满足：

- 用户可以通过登录进入系统，工作台能显示当前用户、组织、空间和角色。
- 个人资料、用户管理、组织管理、空间管理、角色管理和权限策略具备前端入口。
- 一个 MVP 流程可以从创建、配置、启动、暂停、恢复、审核到交付完整跑通。
- 前端至少包含登录、工作台、个人资料、用户/组织/角色管理、工作流列表、编辑器、运行详情和基础资产入口。
- 后端至少包含认证、用户、组织、角色、权限、工作流定义、运行状态机、变量系统、审计日志和基础交付 API。
- 核心逻辑有测试覆盖，尤其是状态机、变量解析、权限判断、输出校验和重试。
- 本地开发命令和必要环境变量在文档中可复现。

## 5. 每次任务的交付要求

- 开始前确认所属阶段和任务编号。
- 修改范围应尽量贴近当前任务，不顺手做无关重构。
- 新增或修改接口时同步契约、文档和测试。
- 新增或修改复杂逻辑时同步补充中文注释；静态数据、模拟交互和后续要接 API 的位置必须标明替换方向。
- 完成后运行对应验证，并在最终说明中写明验证结果。
- 若发现计划与实现冲突，先更新本文档，再继续开发。
